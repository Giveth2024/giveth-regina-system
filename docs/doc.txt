Tables
1. Users
CREATE TABLE IF NOT EXISTS users (
    id varchar(36) NOT NULL PRIMARY KEY,
    full_name VARCHAR(100) NOT NULL,
    username VARCHAR(50) NOT NULL UNIQUE,
    email VARCHAR(100) NOT NULL UNIQUE,
    password VARCHAR(255) NOT NULL, -- Long length to accommodate bcrypt hashes
    phone_number VARCHAR(20),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

CREATE TABLE stock (
  id CHAR(36) NOT NULL,
  item_name VARCHAR(100) UNIQUE NOT NULL,
  item_type ENUM('Direct Sale', 'Kitchen') NOT NULL,
  category VARCHAR(50) NOT NULL,
  barcode VARCHAR(50) UNIQUE DEFAULT NULL,
  unit_type VARCHAR(20) NOT NULL, -- Can be glass, can, plastic
  full_quantity DECIMAL NOT NULL,
  empty_quantity DECIMAL DEFAULT NULL,
  low_stock_alert_level DECIMAL DEFAULT NULL,
  cost_price DECIMAL(10, 2) NOT NULL,
  selling_price DECIMAL(10, 2) NOT NULL,
  expected_profit DECIMAL(10, 2) NOT NULL,
  deleted_at TIMESTAMP DEFAULT NULL,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  KEY idx_products_barcode (barcode),
  KEY idx_products_deleted_at (deleted_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

CREATE TABLE `sales` (
  `id` VARCHAR(36) NOT NULL,
  `user_id` VARCHAR(36) NOT NULL,
  `debtor_id` VARCHAR(36) DEFAULT NULL,,
  `creditor_id` VARCHAR(36) DEFAULT NULL,,
  `invoice_number` VARCHAR(25) UNIQUE NOT NULL,
  `total_amount` DECIMAL(10, 2) NOT NULL DEFAULT 0.00,
  `amount_paid` DECIMAL(10, 2) NOT NULL DEFAULT 0.00,
  `change_amount` DECIMAL(10, 2) NOT NULL DEFAULT 0.00,
  `payment_type` ENUM('Cash', 'Mobile Money', 'Card', 'Debt', 'Credit') NOT NULL DEFAULT 'Cash',
  `created_at` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  CONSTRAINT `fk_sales_user` 
    FOREIGN KEY (`user_id`) REFERENCES `users` (`id`) 
    ON DELETE RESTRICT ON UPDATE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci;

CREATE TABLE `sale_items` (
  `id` VARCHAR(36) NOT NULL,
  `sale_id` VARCHAR(36) NOT NULL,
  
  -- We change this to CHAR(36) and explicitly override its collation 
  -- to match the existing 'stock' table primary key exactly.
  `stock_id` CHAR(36) CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci NOT NULL,
  
  `quantity` INT NOT NULL CHECK (`quantity` > 0),
  `selling_price` DECIMAL(10, 2) NOT NULL,
  `glass_fine_per_unit` DECIMAL(10, 2) NOT NULL DEFAULT 0.00 COMMENT 'Fines charged if bottles are taken away',
  glass_status VARCHAR(20) DEFAULT NULL;
  `subtotal` DECIMAL(10, 2),
  profit DECIMAL(10,2) NOT NULL DEFAULT 0.00,
  PRIMARY KEY (`id`),
  
  -- Links to sales (utf8mb4_0900_ai_ci)
  CONSTRAINT `fk_sale_items_sale` 
    FOREIGN KEY (`sale_id`) REFERENCES `sales` (`id`) 
    ON DELETE CASCADE ON UPDATE CASCADE,
    
  -- Links to stock (utf8mb4_unicode_ci)
  CONSTRAINT `fk_sale_items_stock` 
    FOREIGN KEY (`stock_id`) REFERENCES `stock` (`id`) 
    ON DELETE RESTRICT ON UPDATE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci;

2. Authentication

Login

POST /login

Description
Authenticates an existing user using either their username or email address together with their password. The system validates the request, verifies the user's credentials, generates a JSON Web Token (JWT), stores it in a secure HTTP-only cookie, and returns the authenticated user's information.

Processing Flow

Receive the login request.
- Validate that the required fields (identifier and password) are provided and that no unexpected fields are included.
- Search for a user using the supplied username or email address.
- If no matching user is found, return an "Invalid credentials" response.
- Compare the supplied password with the stored bcrypt password hash.
- If the password is incorrect, return an "Invalid credentials" response.
- Generate a JWT containing the user's ID, username, and email.
- Sign the JWT using the application's secret key.
- Store the JWT in a secure HTTP-only cookie.
- Return a successful authentication response.

Request Body

json
{
    "identifier": "johndoe",
    "password": "Password123"
}


> Note:The identifier field accepts either the user's usernameor email address.

Success Response

Status Code:200 OK

Response Body

{
    "success": true,
    "message": "Login successful!!",
    "data": {
        "full_name": "John Doe"
    }
}

Stock Management
1. Adding Stock

POST/api/giveth/stock/add

--> Description
Adds a new stock item to the inventory. The system validates all required fields, ensures the item type is valid, checks that quantity and pricing fields are numeric, calculates the expected profit, generates a UUID for the stock item, and stores the record in the database.

--> Processing Flow
- Receive the stock creation request.
- Validate that all required fields are provided.
- Validate that the item_type is either Direct Sale - or Kitchen.
- Validate that quantity and pricing fields contain valid numeric values.
- Generate a UUID for the stock ID.
- Calculate the expected profit using:
  Expected Profit = Selling Price - Cost Price
- Insert the stock item into the database.
- Return a successful response.

--> Request Body

{

    "empty_quantity": 0,
    "low_stock_alert_level": 10,

    "selling_price": 1000
}


Notes
- item_type must be either "Direct Sale" or "Kitchen".
- barcode may be null if the item does not have a barcode.
- item_name must be unique.
- If a barcode is provided, it must be unique.
- expected_profit is calculated automatically and should not be included in the request.

2. Deleting Stock
DELETE /api/giveth/stock/:id

--> Description
Performs a soft delete on an existing stock item in the inventory by its unique identifier (id). Instead of permanently destroying the database record, the system updates the deleted_at timestamp column to preserve data integrity for historical auditing, sales tracking, and reporting.

--> Processing Flow
* Receive the soft delete request containing the stock item id as a URL parameter.
* Execute an update query targeting rows where the id matches and the deleted_at field is currently NULL.
* Update the deleted_at field with the current system timestamp (NOW()).
* Evaluate the result set payload:
* If affectedRows equals 0, recognize that the stock item either does not exist or has already undergone a soft delete, and halt the execution process.
* Return a successful completion response confirming the soft deletion.

--> Request Parameters
id (string, Required): The 36-character UUID string belonging to the stock item you intend to remove.

--> Success Response
Status Code:200 OK
Response Body
json
{
    "success": true,
    "message": "Stock with id:8a0944be-aca4-4a0d-9957-8ebe3427a2a6 deleted successfully"
}

--> Error Responses

json

3. Updating Stock Item Details
--------------------------------------------------------------------------------
HTTP Method / Endpoint:
PUT /api/giveth/stock/:id

--> Description
Updates the attributes of an existing active stock item. The endpoint explicitly
validates the provided request body fields and checks data types. If an attribute 
is passed as null, the database retains its pre-existing value using the 
COALESCE SQL pattern. The expected profit is automatically calculated based on 
the incoming pricing data or falls back to database records if one or both 
prices are unchanged.

--> Processing Flow
1. Parse the stock item UUID from the request parameters.
2. Validate that structural fields inside the request body are structurally intact.
3. Validate numeric integrity: Loop through the quantities and pricing values 
   (full_quantity, empty_quantity, low_stock_alert_level, cost_price, selling_price). 
   If a field is not a number and not null, reject the execution.
4. Verify database existence: Query the 'stock' table to verify that an active row 
   exists matching the provided ID (where deleted_at IS NULL).
5. Compute Pricing metrics:
   - If cost_price or selling_price are explicitly null, fetch and utilize the 
     values currently on record in the database.
   - If pricing conditions are explicitly passed, calculate the updated margin:
     Expected Profit = Final Selling Price - Final Cost Price
6. Execute a unified parameterized SQL UPDATE statement utilizing COALESCE on all 
   mappable columns to selectively replace values.
7. Return a successful 200 OK structural response.

--> Request Body Sample (JSON)
{
    "item_name": "ICE - Springs Packaged Drinking Water 1.5Litres",
    "item_type": "Direct Sale",
    "category": null,
    "barcode": null,
    "unit_type": "Plastic",
    "full_quantity": 23,
    "empty_quantity": 0,
    "low_stock_alert_level": 10,
    "cost_price": 833.33,
    "selling_price": 1000
}

Notes:
- Fields explicitly assigned a "null" value will be skipped by the database, 
  retaining their current saved profile metrics.
- The system automatically triggers re-calculation constraints for the 
  expected_profit entry whenever either cost_price or selling_price changes.

--> Success Response
Status Code: 200 OK
Response Body:
{
    "success": true,
    "message": "Stock with id:b62885f0-7d4e-4901-a76b-4ee76932468b updated successfully!!!"
}

--> Error Responses
Status Code: 400 Bad Request
Returned when structurally required parameters fail internal verification.
{
    "success": false,
    "message": "Missing required fields..." 
}

Status Code: 400 Bad Request
Returned if any inventory metric array item contains alpha characters.
{
    "success": false,
    "message": "The value is not a number. Make sure the quantity(both full and empty), low stock alert level, cost price and selling price are numbers"
}

Status Code: 404 Not Found
Returned when the UUID parameter does not map to an active record.
{
    "success": false,

4. Fetching Stock Items
--------------------------------------------------------------------------------
HTTP Method / Endpoint:
GET /api/giveth/stock

--> Description
Retrieves a paginated list of stock items from the inventory. The endpoint supports optional filtering by category, item type, unit type, stock condition, search text, price ranges, quantity ranges, deleted records visibility, and sorting. It returns the matching stock records together with pagination metadata and the filters that were applied.

--> Processing Flow
1. Read the optional query parameters from the request URL.
2. Build a dynamic SQL query that includes only active stock records by default, unless deleted=true is supplied.
3. Append additional filters for category, item_type, unit_type, low stock, out of stock, empty bottle conditions, search terms, price ranges, and quantity ranges.
4. Apply default pagination values when limit or page are missing or invalid.
5. Determine the allowed sort column and order direction; if none is provided, the system defaults to sorting by item_name in ascending order.
6. Add LIMIT and OFFSET clauses to the query.
7. Execute the query and return the matching records with metadata.

--> Query Parameters
- category (string, optional): Filters stock by category.
- item_type (string, optional): Filters stock by item type such as Direct Sale or Kitchen.
- unit_type (string, optional): Filters stock by unit type.
- search (string, optional): Searches by stock id, barcode, or item_name.
- low_stock (boolean, optional): Set to true to return items where full_quantity <= low_stock_alert_level.
- out_of_stock (boolean, optional): Set to true to return items where full_quantity = 0.
- no_empty_bottles (boolean, optional): Set to true to return items where empty_quantity = 0.
- has_empty_bottles (boolean, optional): Set to true to return items where empty_quantity > 0.
- deleted (boolean, optional): Set to true to include soft-deleted records.
- sort_by (string, optional): One of item_name, selling_price, cost_price, full_quantity, empty_quantity, expected_profit, created_at.
- order (string, optional): asc or desc.
- limit (number, optional): Number of records per page. Default is 50.
- page (number, optional): Page number. Default is 1.
- max_selling_price (number, optional): Upper selling price filter.
- min_selling_price (number, optional): Lower selling price filter.
- max_cost_price (number, optional): Upper cost price filter.
- min_cost_price (number, optional): Lower cost price filter.
- max_quantity (number, optional): Upper full quantity filter.
- min_quantity (number, optional): Lower full quantity filter.

5. Adding Sales
--------------------------------------------------------------------------------
HTTP Method / Endpoint:
POST /api/giveth/sales/add

--> Description
Creates a new sale transaction and records each sold item in the sale_items table. The endpoint requires authentication and executes the entire sales workflow inside a database transaction so that the sales record, sale item records, and stock updates are all committed together. If any step fails, the transaction is rolled back to keep the inventory and sales data consistent.

--> Processing Flow
1. Receive the authenticated sales request.
2. Extract the sale-level fields from the request body: user_id, debtor_id, creditor_id, total_amount, amount_paid, change_amount, and items.
3. Generate a UUID for the sale and an invoice number using the current date-based invoice sequence.
4. Insert the sale into the sales table.
5. Validate that the items array is not empty.
6. For each item in the items array:
   - verify that the referenced stock record exists;
   - create a sale_items record with a new UUID and the sale details;
   - update the matching stock record to reduce available quantity.
7. Commit the transaction and return a success response.
8. If any validation, stock check, or stock update fails, roll back the transaction and return an error.

--> Request Body
json
{
    "user_id": "f9c0d6c9-0db4-4d5c-8c7c-91f16a59d3ef",
    "debtor_id": null,
    "creditor_id": null,
    "total_amount": 5000,
    "amount_paid": 5000,
    "change_amount": 0,
    "items": [
        {
            "stock_id": "6b4f0e98-82f3-41d1-aee2-3a4b6c7d8e9f",
            "quantity": 2,
            "selling_price": 2500,
            "glass_fine_per_unit": 0,
            "glass_status": null,
            "subtotal": 5000,
            "profit": 1000
        }
    ]
}

Notes:
- This endpoint requires authentication.
- The items array must contain at least one item.
- Each item must reference an existing stock record.
- The system prevents a sale from exceeding the available stock quantity.
- If any stock update fails, the entire sale is rolled back.

--> Success Response
Status Code: 201 Created
Response Body
json
{
    "success": true,
    "message": "Sales added successfully"
}

--> Error Responses

Returned when the items array is empty.
